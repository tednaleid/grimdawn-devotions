Item-DB Direction: Repo Topology, Data Artifacts, Item Source, grimtools Boundary

date 2026-07-11 topic item-db-direction mode repo-grounded

focus Should the item-database initiative stay in the grimdawn-devotions repo or move to its own? Inputs: size/finality of the gitignored artifacts, whether item "source" (where to get it) is derivable from our data, and the viability of linking out to grimtools for maps/quests.

Grounding Context

Codebase context. One repo currently holds the devotions planner (deployed to GitHub Pages from web/dist only) and the item-data initiative (branch item-data-raw-deposit): the raw deposit, the derived typed schema, committed curation JSON, and a throwaway DuckDB-WASM prototype. The split was pre-engineered to be cheap: deposit_dir is a single justfile variable ("relocating the deposit (including to a separate repo) is a one-line change"), build_deposit.py and build_derived.py share no imports beyond trivial helpers, and CI/deploy never touch item data. The real coupling is upstream: both domains share the Windows-only extraction (just extract, ~5 GB) and the i18n-tables recipe ("the single, generic builder of ALL game text tables").

343 KiB git pack size today (repo is tiny)
20.5 MB data/deposit (gitignored): facts 12.1 + labels 5.7 + meta
~600 KB data/derived (gitignored), 14,130 entities
372 KB devotions.json — the committed-data precedent

Distance to "final" for the derived schema (the honest answer to "how close are we"): four attack-speed tiers marked interpolated; needs one grimtools card oracle, one variance known_gap (scaled offensive lines), and six named Known Gaps in docs/item-schema.md (affix applicability, pet-skill rollup, proc trigger text, unnamed records, plus the two above). Most of the calibration distance closes in one screenshot-harvest evening; the structural gaps ride on the loot-graph work below. meta.parquet already carries game_version, steam_buildid, and row counts — release-tagging provenance exists today.

Acquisition data is already in the deposit (verified by live queries this session): 1,965 loot-table records with lootNameN/lootWeightN edges to items; 4,149 creature records with chanceToEquip* slots (chain verified end-to-end: manticore_h06 → chanceToEquipMisc3=100 → mt_comp_rare_a01); 45 merchant NPC records; 292 factionSource + 128 factionRequired facts; 1,333 container/chest records. One early claim did not survive verification — see the Rejection Summary.

External research (moderate value): grimtools shows "dropped from" only for named-monster sources; vendor, faction, quest, and chest sources are not modeled there, and its item→monster cross-links have documented drift. grimtools item URLs are internal sequential ids — an external site cannot compute item deep links; only name-based search links and /map/areas/<id> are constructible, and grimtools runs per-patch snapshot sites, so even those aren't guaranteed stable. Prior art: GDLoot derives loot tables from lootmastertables + monster equipment (excludes chests); IAGD/GDStash parse database.arz without modeling acquisition. No docs/solutions/ learnings exist; web research was fresh this run.

1

Contracts before topology: stay one repo, write the split tripwires, settle the query IR

repo-topologyConf 85%Cx Medium
Description
Don't decide "one repo or two" now — the codebase already made deferral free. Instead: (a) write down the tripwires that would force a split (first external consumer of the parquet, first independently-deployed item app, item work breaking devotions CI) and stay single-repo until one fires; (b) settle the one genuinely irreversible contract first — the engine-independent, host-independent query IR in the URL hash, because published links are the only artifact you can't migrate; (c) treat the released deposit as the interchange boundary if a split ever happens (it carries all 13 locales' raw text — though display formatting like statFormat.ts conventions stays code, not data).
Basis
direct: justfile:14–16 — "Every deposit recipe resolves through this one variable so relocating the deposit (including to a separate repo) is a one-line change"; docs/deposit.md — "The emitter is self-contained, so moving the deposit to a dedicated item-database repo later stays a cheap option"; BACKLOG query-IR item — "Must be settled before the first shareable item-query link ships." external: data-engineering polyrepo norm: split on divergent consumers/teams, not on domains.
Rationale
Every hard fact found points the same way: deploy ships only web/dist, the builders share no imports, and the only expensive coupling (Windows extraction + i18n-tables) is exactly what a split would duplicate. Meanwhile the long-game feature — a combined gear+devotion character planner, which is grimtools' actual product — is the strongest single argument for one repo: items and devotions share extraction, 13-locale labels, and stat-line formatting.
Downsides
The item work keeps living on a branch of a repo named "devotions", which reads oddly to outsiders; a rename or umbrella framing may eventually be wanted. Tripwires only work if actually written down (BACKLOG) rather than remembered.
2

Data platform in place: lockfile + balance diff in git, parquet in per-buildid GitHub Releases

data-artifactsConf 80%Cx Medium
Description
Never commit the parquet — at any stability. Git permanently holds what git is good at: curation JSON, a ~1 KB deposit.lock manifest (buildid, game version, row counts, sha256 — note: checksum emission is new plumbing; meta.parquet doesn't carry one today), and a machine-generated per-patch balance diff ("Meat Shield offensiveLife 22→24") that is small, delta-diffable, doubles as a pipeline drift smoke test, and is community-valuable content nobody else publishes. The parquet artifacts ship as immutable GitHub Releases tagged by Steam buildid. Locally, just derive writes buildid-namespaced snapshots so a game patch can't clobber the only pre-patch copy.
Basis
direct: docs/deposit.md — "parquet does not delta-diff, so iterating on the deposit's shape while committed would write a full ~18 MB blob into history per format change"; BACKLOG — "Per-patch immutable releases tagged by Steam buildid (already in deposit meta) with checksums and a machine-generated balance diff vs the previous build." external: DVC pointer files, Nix flake.lock, Hugging Face moving datasets off git blobs for the same non-delta reason; OSM publishes the planet file out-of-band while diffs are first-class.
Rationale
This dissolves Ted's "how close to final?" question rather than answering it: finality stops gating anything because releases are versioned and immutable — a schema change is just a new release, and the 343 KiB pack stays tiny forever. It also removes the strongest pro-split argument (keeping blobs out of history) and gives CI and any future second repo a fetchable deposit (idea 1's interchange).
Downsides
Releases are GitHub-specific (portable in principle, but the fetch path hardcodes a host); a fetch-if-missing path in the justfile is new work; the balance-diff generator is a real small project, not a flag flip.
3

Acquisition edges v1: one loot-graph walker, three products, provenance-tagged from day one

item-sourceConf 75%Cx High
Description
Build one transposed (reverse-reachability) walk over the weighted loot graph — creature chanceToEquip* → loot table lootNameN/lootWeightN → item, plus merchant, faction, and quest entry points — emitting rows shaped (item_record, source_record, path_kind, effective_weight). That single artifact answers Ted's "where do I get this item" (drops_from / sold_by / quest_reward edge kinds), delivers the BACKLOG's top deferred item (affix-to-gear applicability), and resolves the 58 blueprints_without_crafts diagnostic. Every emitted edge carries a provenance qualifier — flat-fact | loot-walk | curated-oracle | interpolated — a schema-shape decision that is cheap only if made before the first edge ships.
Basis
direct: BACKLOG — "resolve which prefixes/suffixes can roll on which gear types by walking LootRandomizerTable weights from the gear loot tables. Also resolves the blueprints_without_crafts diagnostic"; verified chain manticore_h06 → chanceToEquipMisc3=100 → mt_comp_rare_a01 (reproduced independently by the verifier). external: search-engine transposed link graphs answer both directions from one index; Wikidata's claim/reference/rank model for the qualifiers; grimtools' documented cross-link drift is the failure the qualifiers prevent.
Rationale
"Source" stops being a new initiative and becomes the already-planned graph work with a second output — one walker amortized across the two most-wanted features plus a known diagnostic. Programmatic derivation with provenance also positions the data to be more correct than grimtools' semi-manual mapping, which demonstrably drifts.
Downsides
The weighted graph has real depth (nested master tables, level-scaled weights, dynamic difficulty tiers) — this is the largest single work item on the list, and effective-weight semantics need care to avoid publishing misleading drop percentages.
4

Tiered source coverage, cheapest tiers first — and it beats grimtools where grimtools is blind

item-sourceConf 65%Cx Medium
Description
Ship source data in tiers ordered by cost, with an explicit "world drop" fallback (the community's own granularity norm): tier 1 = faction/vendor gating (factionSource/factionRequired + 45 merchant inventories) — flat-fact joins, no graph walk; tier 2 = named-monster drops and MIs via the walker (idea 3); tier 3 = chests; tier 4 = quest rewards. Hand-curate domains under ~50 rows (merchants, faction display names, quest wearables) with drift guards instead of building plumbing for them. Tier 1 alone already exceeds grimtools, which models none of vendor/faction/quest/chest acquisition. Caveat from verification: the quest tier's shape is genuinely unknown — the earlier "quest records carry only XP rewards" evidence was refuted (bad SQL; the keys belonged to devotion shrines), and a real perPartyMemberDropItemName item reference was found under storyelements/questitems/ — so tier 4 needs one fresh research pass before it's scoped.
Basis
direct: verified counts — 292 factionSource + 128 factionRequired facts, 45 merchantType records, e.g. merchant FileDescription: "Luther Graves - Merchant Rescue". external: grimtools models "dropped from" only for named-monster sources (background research); GDLoot's deliberate chest exclusion as precedent for tiering; Diablo-community wikis' extract-then-confirm hybrid.
Rationale
It converts "is source a lot of work?" into a cost map: two tiers are cheap, one is the walker, one is a bounded research-then-curate job. And it reframes the feature from catch-up to differentiator — "Sold by Luther Graves (Devil's Crossing), Honored" is an answer grimtools cannot give.
Downsides
Tiered shipping means a period where some items say "world drop" that actually have specific sources — needs honest UI labeling ("source unknown/unresolved") to avoid grimtools-style wrong-link erosion of trust. Quest tier scope is unquantified until the redo pass.
5

Link-out policy: own everything a query can answer; name-search links only, never ids

scope-boundaryConf 85%Cx Low
Description
Adopt one rule for Ted's "link out for maps/quests?" question: anything answerable by a DuckDB query over the deposit (stats, requirements, sources, crafts, factions) is owned natively; only world geometry and lore walkthroughs link out. Mechanically, outbound links must be name-intent links — grimtools name-search URLs or /map/areas/<id>, or wiki name-slug URLs computable from our labels — never grimtools item ids, which are internal sequential values no external site can compute, served from per-patch snapshot hosts. And never link out for item source: once ideas 3–4 land, our source data is strictly better than the incumbent's.
Basis
external: grimtools URL research — item ids not computable externally, per-patch snapshot sites, documented broken cross-links. direct: census — level art (6,826 records), triggervolumes, scriptentities deliberately unmodeled; no outbound links exist anywhere in web/src today (verified: grimtools appears only in design-reference comments).
Rationale
One testable rule settles every future "model X or link out?" debate, including several already implicit in the census. It also forecloses a plausible-sounding plan (deep-link every item card) that would have hit an infeasibility wall mid-implementation.
Downsides
Name-search links inherit a normalization burden (our label text vs grimtools' display strings) and degrade to "search results" rather than landing on the exact page; per-patch snapshot hosts mean even good links rot on grimtools' schedule, not ours.
6

Deploy the evidence: ship /items/ on the existing Pages site with a source-facet spike

repo-topologyConf 60%Cx Medium
Description
Graduate the throwaway browser one notch: publish it at /items/ on the existing GitHub Pages deploy, with the ~600 KB derived parquet, and add a quick source facet (tier-1 flat facts only) to test the feature against real usage. Cost correction from verification: this is not "one workflow edit" — CI runs on ubuntu-latest and cannot regenerate the parquet (extraction is Windows-only), so the derived artifacts must first exist as committed or released files (idea 2 provides exactly that). Also resolve the prototype's CDN-loaded DuckDB against the site's self-contained ethos before shipping.
Basis
direct: deploy.yml uploads only web/dist; itemdb.html ABOUTME — "Throwaway prototype…"; derived output ~600 KB vs the already-shipped 372 KB devotions.json. reasoned: the engine bake-off, the query-IR design, and the topology question all improve with a deployed artifact generating real data instead of speculation.
Rationale
Three open decisions (engine, IR, topology) are currently argued from intuition; a live page converts them to measured questions for one afternoon of work once idea 2's artifact hosting exists. Sequencing note: if facet bitmaps win the bake-off, the derived parquet may remain a build intermediate forever — another reason it never needs a git home.
Downsides
Shipping a rough page under the polished planner's domain has reputational cost (label it beta); depends on idea 2 (artifact fetchable in CI); CDN dependency conflicts with the repo's self-contained deploy habits and needs vendoring or acceptance.
