Grim Dawn Item Data Extraction: a Queryable Dataset as the Foundation

Date 2026-07-03 Topic item-data-extraction Mode repo-grounded
Focus Extract all Grim Dawn item data into a really good, well-known schema (parquet + DuckDB-in-browser as the leading candidate) so it is easily queried and becomes the basis for an item-database website with compound filtering better than grimtools.com/db — and for other future work. Data out first; consider alternatives to the leading candidate.

Grounding Context

Codebase Context. The repo ships a static devotion-planner SPA (TypeScript/Bun, hexagonal core/ports/adapters, GitHub Pages, no backend, URL-hash state, 13-locale i18n through Text descriptors and a Localization port). A Windows-only pipeline (game ArchiveTool.exe on .arz/.arc archives → .dbr key-value files → stdlib-only scripts/parse_devotions.py) already extracts 18,738 item DBR files under extracted/records/ across 26 categories — it just never parses them. The DBR parser discovers keys at runtime (patch-tolerant), expansions auto-discover via the gdx* convention, and just parse already stamps Steam build-id provenance. A shelved --duckdb flag already emits a long-format (dbr, key, value) CSV for devotions. Serving today: devotions.json (372 KB), cover-table.bin (1.8 MB precomputed binary index), per-language game-tag tables, an optional 64 KB Rust WASM module with a TS fallback, and a tuned ~107 KB first load. BACKLOG.md has zero item-database entries — this is greenfield. Item records are sparse and complex: 400+ mostly-zero stat keys per weapon file, min/max damage pairs, multi-boolean applicability flags, and cross-record references to sets, granted skills, and affix pools.

External context (web research, value: high). grimtools.com/db is closed-source; no OSS schema-documented Grim Dawn item dataset exists — that is the open gap. Crate Entertainment ships an official Asset Manager extraction workflow (documented in the Grim Dawn Modding Guide), a friendlier legal posture than PoE/Last Epoch reverse engineering. Cautionary tale: PoE’s RePoE/PyPoE extraction ecosystem forked and rotted for lack of a maintenance owner. Strongest UX analog: Scryfall’s compound query DSL and versioned bulk exports. On engines: DuckDB-WASM + static parquet is a live 2025–26 pattern, but its bundle is heavy (~33 MB eh build) vs sql.js at ~1.5 MB; sql.js-httpvfs does range-request SQLite on GitHub Pages; client-side faceted-search libraries (ItemsJS, FlexSearch) are a legitimate no-engine fallback at tens-of-thousands-of-rows scale.

Past learnings. No docs/solutions/ knowledge base exists; the closest in-repo references are docs/dbr-format.md, docs/assets-and-textures.md, docs/i18n.md, and the dated specs under docs/superpowers/specs/ — one of which explicitly rejected DuckDB/parquet for reachability with an exception clause that arguably describes this exact use case.

1Raw deposit + schema census + query-first spine

axis: pipeline Confidence 90% Complexity Low

Description. Get the data out before designing anything. Run the existing field-agnostic extractor over all 18,738 item DBRs once (the read_dbr reader plus the shelved --duckdb long-format path need only a root-directory change), emitting a lossless (record, key, value) parquet stamped with the Steam build id. That deposit is committed or released, so every schema experiment afterwards runs anywhere — no game install, no ArchiveTool, no “close the game first.” The first derived artifact is a schema census: every distinct key with its type shape, non-zero frequency, cardinality, and per-category coverage. Add justfile DuckDB CLI recipes so the dataset is mined conversationally from day one; the queries actually asked become the requirements log for the eventual filter UI. The census also settles the scope denominator — whether weakly-modeled categories (enemygear/, questitems/, faction/) are in or out — with data instead of debate. i18n ships with the deposit, not after it: because .dbr text fields store tags rather than display text, the facts table is language-free by construction, and a companion labels.parquet of (locale, tag, text) — the full per-locale tag tables for all 13 locales, unfiltered at this layer — makes localized querying a join and the English fallback a COALESCE.

Basis
direct: scripts/parse_devotions.py:745-746 — ap.add_argument("--duckdb", ..., help="Also emit devotion_records.csv (long format) for ad-hoc querying"); write_duckdb_csv at :699 (“Emit a tidy long-format (dbr, key, value) table”); justfile:106-115 game-must-be-closed guard and :151-162 Steam build-id capture; docs/dbr-format.md “Union-wide sparse schema; don’t assume a fixed key set.” External echo: NCBI’s raw-archive-first model in bioinformatics.
Rationale
Every other decision in this document gets cheaper and less speculative once the census exists — it converts “sparse and complex schema” from a fear into a table. The riskiest unknown is not rendering; it is whether the schema answers real compound questions. Mining via SQL before any UI exists discovers which filter capabilities matter at near-zero build cost, and the deposit decouples all of that iteration from the Windows-only bottleneck. This is the highest information-per-hour move available and directly delivers Ted’s stated main goal in its first weekend.
Downsides
The raw long-format deposit for items will be substantially larger than devotions’ 32 MB CSV; it may need release-asset hosting rather than a committed file. A lossless deposit defers, not avoids, the typed-schema work (idea 2).

2Canonical two-table schema: entities + long-form stats, raw stat-id passthrough

axis: item-schema Confidence 80% Complexity Medium

Description. The published, durable shape is a pair: items.parquet (one row per item: id, class, rarity, level, slot, template — the dense low-cardinality columns) and stats.parquet (item_id, stat_id, value_min, value_max — the long form for 400+ mostly-zero stat keys). Stat ids stay raw (offensivePhysicalMin, not an invented affix taxonomy), mirroring how devotion bonuses are already modeled, so labels resolve later through a mapping layer and the stored data never churns. The same fact-table shape can span every bonus-bearer — items, loot affixes, devotion stars, granted skills — making “legendary daggers with attack speed” and “devotions with attack speed” the same query. Website payloads are derived extracts of this canonical pair (the OSM planet-plus-extracts model), exactly as devotions.json (372 KB) and cover-table.bin (1.8 MB) are today derived from a 32 MB raw dump. Named sub-decisions to settle here: min/max pairs as first-class columns, and normalizing sparse boolean flag families (enchant weapon-applicability: axe,1 / dagger,1 / ...) into join tables rather than 15 wide columns.

Basis
direct: docs/dbr-format.md:77-78 — “Bonuses: every key with a non-zero, single-number value… Keep raw stat ids”; extract_bonuses at parse_devotions.py:186-196; the existing 32 MB→372 KB→1.8 MB derivation tiering; enchant flag rows quoted from a01a_enchant.dbr. external: OSM planet-plus-extracts; Wikidata’s entity/statement split.
Rationale
This is the “really good, well-known schema” deliverable itself. The entity+statement split answers the wide-vs-long question for a 400-plus-key sparse domain without a 400-column table, keeps every downstream consumer’s label changes out of the stored data, and cleanly separates the durable interchange artifact (openable in any tool, forever) from whatever the browser ends up loading. Raw-id passthrough is the decision devotions already made and banked; forking a second stat vocabulary now would mean two id systems surviving every patch.
Downsides
Long-form stats make some simple reads (one item’s full sheet) a pivot rather than a row read. The unified cross-entity fact table is a bigger commitment than items-only and could be deferred without losing the core shape.

3Reference-closure edges table: items are a graph, not just rows

axis: item-schema Confidence 80% Complexity Medium

Description. Item records point at other records: itemSetName (the set), itemSkillName (granted skill), modifiedSkillName1..N (skill modifiers), petBonusName, and the loottables/ → lootaffixes/ pools that decide what can roll on what. Ship an explicit edges(src_record, ref_key, dst_record) table alongside the canonical pair, built with the same reference-following logic power_skill_chain already uses for devotion celestial powers. Recursive or join queries then natively answer questions grimtools structurally cannot express as filters: “items whose granted skill deals chaos damage,” “which affixes can roll on this base,” “all members of set Y and what completing it grants.”

Basis
direct: extracted/records/records/items/gearweapons/swords1h/b001_sword.dbr:161 — itemSkillName,records/skills/itemskills/granted/item_summonharbinger.dbr; itemSetName in d013_sword.dbr:169; power_skill_chain reference-following at parse_devotions.py:250-263 with cached lookups. external: MusicBrainz’s relationship (edge) model — the entity tables are boring; the edges are where the interesting queries live.
Rationale
A flat-record extract undersells this data: skill grants, set membership, and affix pools are the difference between a stat spreadsheet and a queryable game model. The emit cost is low (the reference resolver already exists and caches), and it is precisely the class of compound question — across record types — that motivated this whole project.
Downsides
Expands the extraction contract from “the items directory” to its transitive closure (skills, sets, pets), growing scope and the census surface. Recursive queries push toward SQL-capable consumers, which mildly couples to the engine decision (idea 5).

4Dataset-as-product: immutable per-build releases with machine-generated balance diffs

axis: pipeline Confidence 75% Complexity Medium

Description. Treat the dataset — not the website — as the product. Each patch refresh publishes a GitHub Release tagged by the Steam build id the pipeline already captures: parquet, gzipped JSONL, checksums, a schema document, and a machine-generated diff against the previous build (ANTI JOIN in DuckDB: items added, stats changed, values rebalanced). History is never overwritten — point-in-time discipline, because a build Steam no longer serves is unrecoverable if skipped. The diff doubles as data-driven patch notes (Grim Dawn’s official notes are famously incomplete on item numbers — a community hook no other site offers) and as the pipeline’s own drift smoke test: the repo has already shipped one quantified silent-drift bug (49 of 63 celestial powers inflated by a patched assumption), and items multiply that blast radius.

Basis
direct: justfile:151-162 — buildid=$(grep -oE '"buildid"...' "$manifest") stamped for provenance; parse_devotions.py:54-56 — the “inflating ~49 of 63 powers” incident comment. external: Scryfall bulk data (versioned exports seeding an ecosystem of third-party sites — the closest analog to this exact play); point-in-time databases in finance; Ensembl’s numbered releases; RePoE/PyPoE’s rot as the cautionary tale for unowned pipelines.
Rationale
No public, versioned, schema-documented Grim Dawn item dataset exists (verifier note: an unverifiable-but-plausible gap claim from research). Publishing one is a durable contribution independent of any UI, it is the move that compounds hardest — every future tool builds on it instead of re-scraping — and the per-build partition column costs almost nothing now while being impossible to retrofit later.
Downsides
Public artifacts create an implicit maintenance expectation (the RePoE lesson cuts both ways) and a versioning contract third parties pin to. Redistribution of extracted game data is community-normalized and Crate-tolerated rather than explicitly licensed — worth a conscious check before publishing loudly.

5Measure-first engine bake-off: facet bitmaps vs in-browser SQL

axis: storage-query Confidence 85% Complexity Low

Description. Two sound but mutually exclusive runtime architectures came out of ideation; do not pick between them by taste. After the census exists (idea 1), measure real row and byte counts — with non-lootable categories trimmed — and prototype both against the same ten “grimtools can’t do this” queries. The repo has done exactly this before: pre-compressing cover-table.bin was “considered and dropped” when a live check showed the gain didn’t justify the complexity. Notably, the same spec that rejected DuckDB for reachability wrote the exception clause that arguably describes this case: it would “only earn its keep for large analytical queries over much more data.” Whether ~19k items is “much more data” is a measurement, not an argument.

Option A · precomputed facet bitmaps
  • cover-table.bin pattern: canonical item order, one bitmap per facet value, magic header + buildId
  • Compound filters = bitwise AND/OR over Uint32Arrays; microseconds, ~2–3 KB per facet
  • No engine in the critical path; DuckDB stays a build-time tool and power-user escape hatch
  • Cost: derived artifacts to rebuild per release; numeric ranges and free text need extra structures
Option B · DuckDB-WASM range-reads the parquet
  • The canonical parquet is also the runtime file — zero derived artifacts, zero drift
  • Full SQL in the browser: ranges, joins (edges table), free text, arbitrary boolean logic
  • Live 2025–26 pattern (static parquet-over-HTTP with range requests)
  • Cost: ~33 MB engine vs the app’s tuned ~107 KB first load; mobile cold-start
Basis
direct: docs/superpowers/specs/2026-06-21-path-predictor-reachability-design.md:277-281 — DuckDB/parquet “explicitly rejected… would only earn their keep for large analytical queries over much more data” (verified verbatim, including the exception clause); 2026-06-28 first-load spec’s ~107 KB budget and “considered and dropped” precedent; reachWasm.ts graceful WASM-fallback pattern. external: DuckDB-WASM bundle weight vs sql.js (~1.5 MB) and sql.js-httpvfs as the middle option.
Rationale
This is the load-bearing bet of the whole plan, and the two options optimize different goods (first-load discipline vs zero-drift full SQL). The repo’s own precedents supply both the decision criteria and the resolution mechanism: measure, then commit. A hybrid (bitmaps by default, engine lazy-loaded on demand, mirroring reach.wasm’s optional-binary pattern) remains a legitimate outcome of the bake-off.
Downsides
A bake-off costs a week that a gut call would not — but reversing a wrong engine bet after URLs and consumers exist costs far more (see idea 6 for the hedge).

6Engine-independent query IR in the URL hash

axis: filter-url Confidence 65% Complexity Medium

Description. Before the first shareable link ships, define a compact query intermediate representation — facet terms, numeric ranges, text terms, boolean combinators — encoded in the URL hash with the same stale-link tolerance discipline as urlState.ts. A Scryfall-style text DSL (class:dagger rarity:legendary stat:attackSpeed>10) and clickable facet chips are then two front-ends compiling to the same IR, and the loaded engine — facet bitmaps, DuckDB-WASM, sql.js, or a plain JS predicate — is just a compile target. The hash also carries the dataset buildId so a stale link can say “this search ran against build 19149150.”

Basis
direct: web/src/core/urlState.ts:111-118 — the canonical-order bitset scheme built for a small fixed id universe, which would reproduce grimtools’ expressiveness ceiling if reused as-is for items; CLAUDE.md’s bookmarkable-URL invariant. external: Scryfall’s query DSL; IVOA ADQL (astronomy’s portable-query standard) as the looser structural analog.
Rationale
Every shipped link is a compatibility contract. If URLs encode raw SQL or engine-shaped state, the storage bet silently freezes forever; if they encode an IR, engines can be swapped, benchmarked, and lazy-loaded behind stable links. This is also where Ted’s core complaint gets solved structurally: compound, non-mutually-exclusive predicates are native to the IR rather than a UI matrix bolted on later.
Downsides
Designing an IR before the census and query log exist risks abstraction in the wrong places — sequence it after idea 1’s mining phase. Medium confidence: the right IR scope (how much boolean power, whether free text is first-class) is genuinely open until real queries accumulate.

7Exception-only stat-label generator for 13-locale i18n

axis: i18n-tags Confidence 65% Complexity Medium

Description. Items need display labels for 700+ raw stat ids across 13 locales; data/stat-tags.json’s hand-curated 36 entries cannot scale linearly. But the stat-id naming convention is decomposable (offensivePhysicalMin = prefix + domain + modifier, the same families as the parser’s POWER_STAT_PREFIXES), and the game’s own text tables must contain a display string for every stat its tooltips render. Build a generator that proposes a candidate game tag per stat id via name transforms, verifies each resolves in Text_EN, auto-accepts verified hits, and emits only the misses for hand curation — inverting “curate every key” into “curate the exceptions.” Extend the existing statHumanizeCoverage guard-test pattern so unlabeled item stats fail CI instead of silently rendering raw ids to users.

Basis
direct: data/stat-tags.json (36 entries, verified) vs 400+ stat keys per weapon file; POWER_STAT_PREFIXES at parse_devotions.py:71; data/stat-format-tags.json’s "defensiveConvert": "DefenseConvert" — an existing mechanical name transform; statHumanizeCoverage.test.ts as the proven coverage-guard mechanism (docs/i18n.md:287).
Rationale
Label coverage is the item database’s largest grind and its biggest i18n risk — the difference between shipping 13 locales and shipping English with a backlog item. The repo’s i18n invariant makes raw-id rendering a rule violation, so this work is mandatory; the only question is whether it is authorship or verification. The core dataset stays language-free (tags only) per the existing binding invariant; per-locale label tables ride the existing build_game_tables.py machinery scaled up.
Downsides
The transform heuristics are unproven until tried against the real tag tables — the auto-accept rate could be 90% or 40%, and the miss list at 40% is still a real curation project. Verification against Text_EN only proves a tag exists, not that it is the right label for that stat in tooltip context; spot-checking is still needed.
