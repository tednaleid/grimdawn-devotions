# Internationalization

How the planner resolves user-facing text to the viewer's language. For the
original design rationale and phasing decisions, see
[docs/superpowers/specs/2026-06-30-i18n-localization-design.md](superpowers/specs/2026-06-30-i18n-localization-design.md).
This document describes the system as it exists today. All four phases are
done: Phase 1a built the seam and app-owned chrome strings; Phase 1b added
game-data text; Phase 2 mapped a curated set of stat names onto game tags;
Phase 3 extracted the remaining game tables and authored the remaining app
catalogs, and enabled 12 locales in detection and loading; a follow-on added
Spanish as a 13th, and another added a visible language picker. Remaining
follow-up items (community correction of authored translations, and others
noted throughout) are tracked in `BACKLOG.md`.

## Supported locales

Thirteen locales ship today: `en de fr es ru zh pl it cs ja ko pt vi`. Each
has both a game table (`data/i18n/game.<locale>.json`) and an app catalog
(`web/src/i18n/app.<locale>.json`); `SUPPORTED_LOCALES` in
`web/src/adapters/localizationAdapter.ts` is the single list gating both
detection and loading.

Spanish (`es`) shipped after an initial gap: `Text_ES.arc`, the game's
Spanish text archive, first failed to open in ArchiveTool ("Failed to open")
on the machine used for extraction, while all other bundled non-English
languages extracted fine. The cause was a transient file-handle lock on the
archive plus a relative-path extraction bug (see the ArchiveTool gotchas
below); a Steam "verify integrity of game files" pass plus using an absolute
extract path resolved it, and `es` now extracts and builds like every other
language.

## The port

`web/src/ports/Localization.ts` defines the seam `core/` renders through:

```ts
interface Localization {
  translate(key: string, params?: Record<string, string | number>): string;
  gameText(tag: string): string;
  locale: string;
}
```

- `translate(key, params?)` resolves an app-authored catalog key to display
  text in the active locale, with named-placeholder interpolation (`"{count}
  used"`).
- `gameText(tag)` resolves a game-data tag (for example `tagDevotion_A01`)
  against an extracted per-language text table, the same way `translate`
  resolves an app key. `data/devotions.json` carries tags (`name_tag`,
  `description_tag`, ...) rather than baked English, so every constellation,
  celestial power, pet, and weapon-requirement string a view renders goes
  through `gameText`.

`core/` stays language independent: it works with ids and tags, never display
text. Only adapters and view modules call `translate`/`gameText`.

## Resolution: `web/src/core/localization.ts`

`makeLocalization(active, fallback, locale)` builds a `Localization` from two
flat key-to-string maps. `translate` looks a key up with a per-key fallback
chain:

1. the active locale's catalog,
2. then the English catalog,
3. then the raw key itself.

A catalog value of `""` counts as missing for this purpose (an untranslated
key authored as an empty string falls through the chain instead of rendering
blank), so it never throws and never returns blank: a partially translated
language shows English for its missing keys, and a totally broken build shows
the key verbatim rather than an empty string. `setLocalization`/`translate` also hold a
module-level singleton so view code can call `translate(key)` directly without
threading the port through every function signature (mirroring how `core/`
modules are otherwise pure and argument-driven, but this one resolver is
deliberately global, like a logger).

## Locale detection: `web/src/core/locale.ts`

`pickLocale(preferred, available)` is a pure function: given an ordered list of
preferred locales (from the browser) and the set of locales the app ships, it
returns the first preferred locale (base language, e.g. `de` from `de-DE`)
that is available, else `"en"`.

## Loading: `web/src/adapters/localizationAdapter.ts`

`loadLocalization(opts)` is the adapter that ties detection and resolution
together at boot:

1. Reads the preferred-locale list from `navigator.languages` (or an injected
   list, for tests), unless the viewer has an explicit override stored (see
   the language picker below), in which case that locale is passed instead.
2. Calls `pickLocale` against the shipped `available` locales to choose the
   active locale.
3. Fetches `app.en.json` and `data/i18n/game.en.json` (always, as fallbacks)
   and, if the active locale is not English, `app.<locale>.json` and
   `data/i18n/game.<locale>.json`.
4. Builds a `Localization` via `makeLocalization` and installs it as the
   singleton via `setLocalization`.

A fetch failure or malformed JSON degrades to an empty catalog for that
bundle, never a thrown error; the per-key fallback chain absorbs the gap. The
UI never blocks on i18n: `main.ts` awaits `loadLocalization` once during boot,
before rendering anything that calls `translate`.

Locale is a viewer preference, not app state: it is never written to or read
from the URL hash. Selection ids, stat ids, and every other piece of shareable
state stay language independent, so a build link opens correctly in any
viewer's browser regardless of their language.

## Language picker: `web/src/adapters/languagePicker.ts`

A small globe button at the header's right edge (`.lang-picker` in
`styles.css`) opens a compact popover listing the 13 shipped locales by their
endonym (each locale's name in its own language, e.g. "Deutsch", "Русский" -
`LOCALE_NAMES` in `localizationAdapter.ts`, a fixed constant, not a `translate`
key, since an endonym does not change with the active UI language). Selecting
one:

1. Persists the choice via `storeLocale` (`localStorage`, key `locale`) so it
   survives a reload; `storedLocale` reads it back at boot and, when it names a
   supported locale, passes it to `loadLocalization` as the sole `preferred`
   entry, overriding browser auto-detection. An absent or unsupported stored
   value (never written, or a stale/hand-edited one) falls back to
   auto-detection, same as before the picker existed.
2. Calls `loadLocalization` again with the new locale, replacing the
   singleton.
3. Re-applies the static header chrome (`applyChrome()` in `main.ts`) and
   rebuilds the benefit/pet catalogs (`buildCatalogs()`) - `condensedRows`
   resolves each subject's label via `translate`/`gameText` at the time it is
   called, so a catalog built once at boot holds labels frozen in whatever
   locale was active then; a language switch must rebuild it or the "Available
   to get" panel keeps showing the old language after everything else has
   switched.
4. Calls `refresh()`, which re-renders every other language-dependent view
   (sidebars, tooltips, build order) since those already recompute from
   `translate`/`gameText` on every render rather than caching a result.

The map's SVG itself carries no text (constellation art is language
independent; names render only in tooltips/sidebars), so no map rebuild is
needed on a switch.

## The catalog artifacts

Three separate committed artifacts, each with a different owner and lifecycle:

1. **`web/src/i18n/app.<locale>.json`** (authored now, Phase 1a). Flat
   `{ key: text }` maps for app-owned strings that have no game source: UI
   chrome (header, drawer, tooltip, build order, commit buttons) and the
   `statFormat` labels (the app's own English names for the game's internal
   stat ids, e.g. `stat.attr.Strength` -> "Physique"). English
   (`app.en.json`) is authored directly; other locales are authored
   translations of the same keys, and are optional (missing keys fall back to
   English).
2. **`data/i18n/game.<locale>.json`** (extracted, Phase 1b, done; builder
   split out in Phase 2). A `{ tag: text }` table per language, filtered to
   only the tags `data/devotions.json` and `data/stat-tags.json` reference,
   extracted from the game's own `Text_XX.arc` files.
   `scripts/parse_devotions.py` writes the tag (not the resolved text) into
   `devotions.json` and, for validation only, keeps its own in-memory
   tag-to-English table to assert every referenced tag resolves; it no
   longer writes `game.<locale>.json` itself.
   `scripts/build_game_tables.py` builds the table: it collects every
   `*_tag` value out of `devotions.json` plus every tag value in
   `data/stat-tags.json`, resolves each against a language's extracted text
   directory, and writes `game.<lang>.json`. It reads `devotions.json` rather
   than re-deriving ids from source records, so it never touches the parser's
   `slugify(English name)` constellation/star ids, and can run once per
   language (`--lang`, `--text-dir`) without the parser in the loop; a tag
   that does not resolve in a language is simply omitted; `gameText` falls
   back to English for it at runtime. This is the authoritative source for
   constellation names, celestial power names and descriptions, weapon
   requirements, pet names, and (Phase 2) the stat names in
   `data/stat-tags.json`. Racial targets and the proc trigger word are
   app-catalogued instead (`stat.race.*`, `trigger.<enum>` in
   `app.<locale>.json`), since they come from a small closed enum rather than
   free game text.
3. **`data/stat-tags.json`** (curated, Phase 2, done). A language-independent
   map from 36 `statFormat` catalog keys (`stat.attr.*`, `stat.damage.*`,
   `stat.dot.*`, `stat.resist.*` - attributes, instant damage types,
   damage-over-time types, and resistances) to a canonical game tag, for stats
   where a clean in-game term exists (e.g. `stat.dot.Cold` ->
   `tagCharStatsDurFrostburn`, "Frostburn"). Each mapping was chosen so the
   game's English text is byte-identical to the app's existing authored
   English for that key, and cross-checked that the tag also resolves in the
   German table; ambiguous or dubious matches were rejected and left
   app-authored. `web/src/core/statTags.ts` imports the JSON directly, so the
   map is bundled into the JS (no runtime fetch, no dist copy step). Mapped
   stats resolve via `gameText(tag)` so the exact translation comes from the
   game itself rather than an authored guess; unmapped stats keep their
   `app.<locale>.json` entry.
4. **`data/stat-format-tags.json`** (curated). A language-independent map from a
   raw GD stat id to a game tag whose display text is a *value-embedded format
   string* (`"{v}% <noun>"`, e.g. `defensivePercentCurrentLife` ->
   `DefensePercentCurrentLife` = "{%.0f0}% Resistance to Life Reduction"). Unlike
   `stat-tags.json`, whose tags are bare nouns keyed by a stat catalog key, these
   are keyed by the raw stat id (they are reached from `classify`'s fallback, not a
   catalog key) and their game text carries the value placeholder. `statFormat`
   resolves them via `gameText` and strips the leading value token
   (`stripValueTokens`) to recover the bare noun; the numeric value renders
   separately. This is used only for value-PREFIX formats (the value leads the noun
   in every language, so a strip localizes cleanly). A value-SUFFIX stat
   (`characterHealIncreasePercent`, "Healing Effects Increased by {v}%") cannot
   reduce to a clean prefix label and is app-authored via `stat.override.<id>`
   instead. `build_game_tables.py` includes these tags in every `game.<lang>.json`
   (via its `--stat-format-tags` argument).

## What's localized today

Every string added since Phase 1a started resolves through `translate` or
`gameText`: the `<head>`/header/sidebar/drawer chrome (including the drawer
button labels and the point-budget slider's `aria-label`), the
constellation/star tooltip, the build-order panel, the commit (add/remove)
buttons, the benefit rows and section headers, and the affinity names all go
through `translate`. `web/test/appCatalog.test.ts` guards the `translate`
side: it asserts every key referenced by the app exists in `app.en.json`, so
a missing key fails CI, not runtime.

One exception, intentional: `web/index.html`'s pre-bundle boot markup (the
loading spinner text and the inline `bootFailed()` recovery script) is plain
English in the HTML shell, because it can render before any JS bundle (and
therefore any catalog) has loaded. `main.ts` overwrites the rest of the static
shell's text (header labels, drawer button labels, document title, the slider
`aria-label`) with `translate()` calls as part of boot, before the user
interacts with anything.

Game-data text is resolved through `gameText`: the constellation/star tooltip,
build-order panel, and celestial-power display all call `gameText(tag)` on the
`name_tag`/`description_tag` fields from `data/devotions.json`. The proc
trigger word (the "on Attack" / "on Critical Hit" part of a celestial power's
tooltip) resolves through `translate("trigger.<enum>")` against the app
catalog instead, since the trigger is one of a small closed set of enums, not
free game text.

`web/src/core/statFormat.ts` mixes both: its internal `statLabel(key)` helper
looks the key up in `STAT_TAGS` (`web/src/core/statTags.ts`, backed by
`data/stat-tags.json`) and calls `gameText(tag)` when a mapping exists,
otherwise falls through to `translate(key)`. The ~36 mapped keys (attributes,
instant damage types, damage-over-time types, resistances) render the exact
in-game term; everything else - section headers, the `stat.template.*`/
`stat.power.*` composed phrases ("N% Chance of X"-style ability lines), the
`stat.subject.*`/`stat.override.*` one-off and irregular stats, and pet
labels - is app-authored via `translate`, because it has no single clean game
tag to point at (a template, a composed phrase, or an ambiguous/irregular
term).

Celestial-power crowd-control and status procs (`formatPowerStats`) render as
composed ability lines rather than raw stat ids: Stun/Freeze/Petrify/Knockdown/
Confusion as "X% Chance of N Seconds of `<Effect>`" (the `stat.power.cc*`
templates), and fumble/slows/resistance-reduction debuffs via the
`timedDebuffs` block; both reuse the condensed view's `stat.subject.*`
vocabulary for the effect name.

No user-facing stat is left to the mechanical `humanize()` fallback. That is a
maintained invariant, guarded by `web/test/statHumanizeCoverage.test.ts`: it
walks every stat id in `data/devotions.json`, renders each through the real
view functions (`groupedBonusRows`, `formatPowerStats`), and fails if any label
is a raw humanization. A new expansion's stat that no rule covers therefore
fails CI rather than shipping as gibberish.

## How to add or refresh a language

1. Run `just i18n-tables` (Windows-only, needs ArchiveTool and a local Grim
   Dawn install) to build `data/i18n/game.<lang>.json` for every installed
   language via `scripts/build_game_tables.py`, which reads
   `data/devotions.json`, `data/stat-tags.json`, and `data/stat-format-tags.json`
   for the tags to resolve and the extracted text for the translations. This is the single generic
   builder for ALL game tables, English included: English reuses the merged
   `extracted/text_en` that `just extract` produces (`just parse` no longer
   builds `game.en.json` itself), while every other language is extracted
   per-arc from `resources/Text_<LANG>.arc`. It auto-discovers every
   installed `Text_*.arc` with no recipe change needed for a language Crate
   adds later; pass explicit language codes to build a subset, e.g.
   `just i18n-tables es fr`. An arc that fails to extract is skipped with a
   message rather than failing the whole run. This is also how you refresh
   an existing language's game table after a
   `data/devotions.json`/`data/stat-tags.json` change.

   ArchiveTool gotchas (the `i18n-tables` recipe already handles both; only
   matters if you invoke ArchiveTool by hand): give `-extract` an ABSOLUTE
   output path. A relative path resolves against ArchiveTool's own working
   directory, not yours, and it fails to open the output file - it prints its
   progress and exits 0 but writes zero files, and on a debug build pops an
   `archivewriter.cpp` assert (`outputFile != INVALID_HANDLE_VALUE`). Also
   redirect stdin (`< /dev/null`); ArchiveTool blocks waiting on stdin in a
   non-interactive shell.
2. Copy `web/src/i18n/app.en.json` to `web/src/i18n/app.<locale>.json` and
   translate the values (keys must not change).
3. Add `<locale>` to `SUPPORTED_LOCALES` in
   `web/src/adapters/localizationAdapter.ts`.
4. Reorder your browser's language preferences (or pass `preferred` to
   `loadLocalization` in a test) to exercise `pickLocale` choosing the new
   locale.
5. Partial translation is fine: any key missing from the new catalog falls
   back to English automatically.

See the language picker section above for the in-app override control; see
`BACKLOG.md` for remaining items (community correction of authored
translations).
