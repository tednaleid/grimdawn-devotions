# Internationalization

How the planner resolves user-facing text to the viewer's language. For the
original design rationale and phasing decisions, see
[docs/superpowers/specs/2026-06-30-i18n-localization-design.md](superpowers/specs/2026-06-30-i18n-localization-design.md).
This document describes the system as it exists today (Phase 1a: the seam and
app-owned chrome strings); Phase 1b and later add game-data text.

## The port

`web/src/ports/Localization.ts` defines the seam `core/` renders through:

```ts
interface Localization {
  translate(key: string, params?: Record<string, string | number>): string;
  locale: string;
}
```

- `translate(key, params?)` resolves an app-authored catalog key to display
  text in the active locale, with named-placeholder interpolation (`"{count}
  used"`). This is the only resolution method today.
- `gameText(tag)` is planned for Phase 1b. It will resolve a game-data tag
  (for example `tagDevotion_A01`) against an extracted per-language text table,
  the same way `translate` resolves an app key. It has no consumer yet because
  no game data carries tags until the parser preserves them (Phase 1b); adding
  it to the interface then is non-breaking.

`core/` stays language independent: it works with ids and tags, never display
text. Only adapters and view modules call `translate`/`gameText`.

## Resolution: `web/src/core/localization.ts`

`makeLocalization(active, fallback, locale)` builds a `Localization` from two
flat key-to-string maps. `translate` looks a key up with a per-key fallback
chain:

1. the active locale's catalog,
2. then the English catalog,
3. then the raw key itself.

It never throws and never returns blank: a partially translated language shows
English for its missing keys, and a totally broken build shows the key
verbatim rather than an empty string. `setLocalization`/`translate` also hold a
module-level singleton so view code can call `translate(key)` directly without
threading the port through every function signature (mirroring how `core/`
modules are otherwise pure and argument-driven, but this one resolver is
deliberately global, like a logger).

## Locale detection: `web/src/core/locale.ts`

`pickLocale(preferred, available)` is a pure function: given an ordered list of
preferred locales (from the browser) and the set of locales the app ships, it
returns the first preferred locale (base language, e.g. `de` from `de-DE`)
that is available, else `"en"`.

## Loading: `web/src/adapters/localizationAdapter.ts`

`loadLocalization(opts)` is the adapter that ties detection and resolution
together at boot:

1. Reads the preferred-locale list from `navigator.languages` (or an injected
   list, for tests).
2. Calls `pickLocale` against the shipped `available` locales to choose the
   active locale.
3. Fetches `app.en.json` (always, as the fallback) and, if the active locale
   is not English, `app.<locale>.json`.
4. Builds a `Localization` via `makeLocalization` and installs it as the
   singleton via `setLocalization`.

A fetch failure or malformed JSON degrades to an empty catalog for that
bundle, never a thrown error; the per-key fallback chain absorbs the gap. The
UI never blocks on i18n: `main.ts` awaits `loadLocalization` once during boot,
before rendering anything that calls `translate`.

Locale is a viewer preference, not app state: it is never written to or read
from the URL hash. Selection ids, stat ids, and every other piece of shareable
state stay language independent, so a build link opens correctly in any
viewer's browser regardless of their language.

## The catalog artifacts

Three separate committed artifacts, each with a different owner and lifecycle:

1. **`web/src/i18n/app.<locale>.json`** (authored now, Phase 1a). Flat
   `{ key: text }` maps for app-owned strings that have no game source: UI
   chrome (header, drawer, tooltip, build order, commit buttons) and the
   `statFormat` labels (the app's own English names for the game's internal
   stat ids, e.g. `stat.attr.Strength` -> "Physique"). English
   (`app.en.json`) is authored directly; other locales are authored
   translations of the same keys, and are optional (missing keys fall back to
   English).
2. **`data/i18n/game.<locale>.json`** (extracted, Phase 1b). A `{ tag: text }`
   table per language, filtered to only the tags `data/devotions.json`
   references, extracted from the game's own `Text_XX.arc` files. This is the
   authoritative source for constellation names, celestial power names and
   descriptions, racial targets, weapon requirements, and pet names, once the
   parser preserves tags instead of baking English into the dataset.
3. **`data/stat-tags.json`** (curated, Phase 2). A language-independent map
   from the app's internal stat ids to a game tag, for stats where a clean
   in-game term exists (e.g. "Frostburn"). Mapped stats resolve via
   `gameText(tag)` so the exact translation comes from the game itself, rather
   than an authored guess; unmapped stats keep their `app.<locale>.json` entry.

## What's localized today

Every string added since Phase 1a started resolves through `translate`: the
`<head>`/header/sidebar/drawer chrome, the constellation/star tooltip, the
build-order panel, the commit (add/remove) buttons, the benefit rows, the
affinity names, and every `statFormat` label. `web/test/appCatalog.test.ts`
guards this: it asserts every key referenced by the app exists in
`app.en.json`, so a missing key fails CI, not runtime.

Two exceptions, both intentional: `web/index.html`'s pre-bundle boot markup
(the loading spinner text and the inline `bootFailed()` recovery script) is
plain English in the HTML shell, because it can render before any JS bundle
(and therefore any catalog) has loaded. `main.ts` overwrites the rest of the
static shell's text (header labels, drawer button labels, document title) with
`translate()` calls as part of boot, before the user interacts with anything.

Game-data text (constellation and power names, descriptions) is still baked
into `data/devotions.json` as English and is out of scope until Phase 1b.

## How to add a language

1. Copy `web/src/i18n/app.en.json` to `web/src/i18n/app.<locale>.json` and
   translate the values (keys must not change).
2. Add `<locale>` to the `available` list passed to `loadLocalization` in
   `web/src/app/main.ts`.
3. Reorder your browser's language preferences (or pass `preferred` to
   `loadLocalization` in a test) to exercise `pickLocale` choosing the new
   locale.
4. Partial translation is fine: any key missing from the new catalog falls
   back to English automatically.

A visible in-app language picker does not exist yet (detection is automatic
from the browser); see `BACKLOG.md` for that and the remaining i18n phases.
